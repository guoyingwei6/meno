# Meno 语音笔记设计

日期：2026-04-13

## 目标

为 meno 增加语音笔记能力，在编辑框图片按钮前新增语音按钮，支持：

- 网页端录音
- 录音结束后预览、重录、取消、保存
- 保存语音文件为 memo 的结构化附件
- 语音笔记先可见，再异步补转写
- 转写结果完成后作为 memo 正文展示

一期优先目标：

- 先把“录音并保存音频笔记”做稳
- 转写采用可降级方案，不阻塞保存

明确不属于一期的范围：

- 导出语音笔记
- 波形图
- 录音中实时转写
- 复杂后台上传
- 重新识别入口
- 服务端强依赖的稳定转写能力

## 产品定义

语音笔记仍然是 memo，只是额外带有一份结构化的语音信息。

- `memo.content` 仍然是主正文字段
- 音频文件作为 memo 的结构化附件存在
- 转写文本是从语音派生出的正文来源之一

一期的语义约定：

- 用户如果只录音不输入文字，创建时允许 `content` 为空
- memo 创建成功后，时间线先显示音频播放器
- 如果转写尚未完成，正文区域可以为空，或显示“语音已保存，等待转写”
- 转写完成后：
  - `voiceNote.transcriptText` 保存转写内容
  - 如果该 memo 原本 `content` 为空，则将转写文本回填进 `content`
  - 如果用户已经手写正文，则保留现有 `content`，转写文本只保留在语音字段中，后续再决定是否提供“插入正文”能力

这样做的目的：

- 不破坏现有围绕 `content` 的列表、搜索、详情、标签、RAG 逻辑
- 允许“音频先保存，文字后到”
- 为后续“重新识别”“不同转写来源”“导出”保留结构空间

## 交互设计

### 编辑器入口

在 `MemoComposer` 工具栏中，将语音按钮放在图片按钮之前。

按钮状态：

- 默认：普通工具按钮
- 录音中：强调态，颜色明显变化，提示当前正在录音
- 不支持录音：隐藏或禁用，并提示当前浏览器不支持录音

### 编辑器状态机

语音录制流程分为四个状态：

1. `idle`
   - 默认状态，仅显示语音按钮
2. `recording`
   - 用户授权麦克风后开始录音
   - 显示录音状态、录音时长、停止、取消
3. `review`
   - 停止录音后展示本地音频预览
   - 提供 `重录 / 保存 / 取消`
4. `saving`
   - 点击保存后先上传音频，再创建 memo
   - 成功后清空录音状态并恢复普通编辑器

### 桌面网页端 UI

桌面端采用紧凑横向布局：

- 语音按钮和图片按钮同级
- 录音中时，在编辑框下方插入一条语音面板，不使用全屏遮罩
- `recording` 态横向展示：
  - 状态点
  - 录音时长
  - 停止
  - 取消
- `review` 态横向展示：
  - 音频播放器
  - 重录
  - 保存
  - 取消

视觉方向：

- 不做花哨动效，重点是清晰和可靠
- 录音态颜色要有明显变化，但不能偏离现有暗色/浅色卡片体系
- 播放器区域应比正文更有边界感，接近参考图中的深色条状块

### 手机网页端 UI

手机端采用纵向堆叠布局，优先保证可点按面积：

- 语音按钮仍位于图片按钮前
- 点击后在 textarea 下方插入整块语音面板
- `recording` 态优先展示：
  - 录音时长
  - 停止
  - 取消
- `review` 态展示：
  - 本地播放器
  - 重录
  - 保存
  - 取消

移动端约束：

- 不依赖 hover
- 不把多个小按钮强塞进一行
- 面板按钮允许换行
- 避免键盘弹起时布局抖动

### 保存后的卡片展示

时间线卡片和详情页统一遵循：

- 音频播放器显示在正文上方
- 转写完成前：
  - 若 `content` 非空，先显示现有正文
  - 若 `content` 为空，显示空态文案或不显示正文
- 转写完成后：
  - 正文区域正常显示 `content`
  - 若 `content` 来自转写，用户感知上就是普通笔记正文

一期不做：

- “展开/收起转写”
- “重新识别”
- “下载音频”

但展示结构需预留后续扩展位置。

## 数据模型

推荐采用“主 memo 表 + 独立语音表”方案，而不是把语音状态直接塞进 memo 主表。

### 主表保持不变的原则

`memos` 继续承载通用字段：

- `id`
- `slug`
- `content`
- `visibility`
- `displayDate`
- 现有时间字段和状态字段

这样可以避免普通 memo 查询被语音字段污染。

### 新增语音表

新增 `memo_voice_notes`，一对一关联 memo。

建议字段：

- `id`
- `memo_id`
- `asset_id` 或 `object_key`
- `audio_url`
- `mime_type`
- `duration_ms`
- `transcript_status`
- `transcript_text`
- `transcript_source`
- `transcript_error`
- `transcript_attempts`
- `transcript_started_at`
- `transcript_completed_at`
- `created_at`
- `updated_at`

`transcript_status` 取值建议：

- `pending`
- `processing`
- `done`
- `failed`
- `not_available`

说明：

- `pending`：等待异步处理
- `processing`：任务处理中
- `done`：转写成功
- `failed`：尝试过但失败
- `not_available`：当前环境没有可用转写能力

### 前端类型扩展

在共享类型中为 memo 增加可选字段：

- `voiceNote?: { ... }`

包含最少这些属性：

- `audioUrl`
- `mimeType`
- `durationMs`
- `transcriptStatus`
- `transcriptText`
- `transcriptSource`

这样前端渲染层可以明确知道一条 memo 是否为语音 memo，而不是再从正文 markdown 中猜测。

## 创建与异步处理流程

### 创建流程

1. 用户点击语音按钮开始录音
2. 结束录音后进入 review 态
3. 用户点击保存
4. 前端上传音频到现有 `POST /api/uploads`
5. 上传成功后调用创建 memo 接口，提交：
   - `content`
   - `visibility`
   - `displayDate`
   - `voiceNote`
6. 后端创建 memo
7. 后端创建对应 `memo_voice_notes` 记录，状态初始化为：
   - 有可用转写能力时：`pending`
   - 无可用转写能力时：`not_available`
8. 后端立即返回成功
9. 时间线刷新后展示带播放器的 memo

### 异步转写流程

转写不应放在创建 memo 的同步请求里。

推荐流程：

1. 定时任务或队列消费者扫描 `pending` 语音记录
2. 将状态置为 `processing`
3. 调用当前可用的转写能力
4. 成功时写入：
   - `transcript_text`
   - `transcript_status = done`
   - `transcript_source`
   - `transcript_completed_at`
5. 若 memo 的 `content` 为空，则把转写文本回填到 `memos.content`
6. 失败时写入：
   - `transcript_status = failed`
   - `transcript_error`
   - `transcript_attempts + 1`

### 一期转写能力策略

一期采用“可降级”方案：

- 若浏览器存在原生语音识别能力，可作为增强手段提供初步转写
- 若浏览器不支持，则不影响录音保存
- 服务端异步转写能力可以先保留队列和状态结构，具体实现可后补

一期的关键原则：

- 转写永远不能阻塞 memo 创建
- 没有转写能力时也必须能保存语音 memo
- 转写失败也不影响音频 memo 被正常查看和播放

## API 设计方向

### 上传

继续复用现有上传接口：

- `POST /api/uploads`

无需为音频单独设计完全独立的上传链路，一期只需要在前端允许上传音频 blob。

### 创建 memo

扩展 `POST /api/memos` 请求体，新增可选 `voiceNote`：

```json
{
  "content": "",
  "visibility": "public",
  "displayDate": "2026-04-13",
  "voiceNote": {
    "audioUrl": "https://...",
    "objectKey": "uploads/...",
    "mimeType": "audio/webm",
    "durationMs": 7000
  }
}
```

更新 memo 时也要允许保留该字段，但一期不要求在编辑页里编辑语音。

### 查询 memo

列表和详情接口返回的 memo 结构中增加可选 `voiceNote` 字段，用于前端直接渲染播放器和转写状态。

## 兼容性与降级

### 录音能力

录音优先基于浏览器原生 `MediaRecorder`。

兼容处理：

- 支持时启用录音按钮
- 不支持时隐藏或禁用，并给出轻提示

### 转写能力

一期不把浏览器原生识别视为硬依赖。

处理策略：

- 能识别则用
- 不能识别则只保留音频
- 服务端异步转写未接入时，语音记录也应可长期处于 `not_available`

## 错误处理

需要明确处理以下失败场景：

- 用户拒绝麦克风权限
- 浏览器不支持 `MediaRecorder`
- 录音中断
- 上传失败
- 创建 memo 失败
- 异步转写失败

对应原则：

- 麦克风权限失败：给出明确提示，不改变编辑器正文
- 上传失败：保留 review 态，允许重试保存
- 创建 memo 失败：保留 review 态，允许重新提交
- 转写失败：memo 仍可查看和播放，只更新转写状态

## 测试策略

一期至少需要覆盖这些测试。

### 前端

- 语音按钮在工具栏中显示在图片按钮前
- 录音状态机从 `idle -> recording -> review -> saving` 正常切换
- review 态保存时先上传音频，再创建 memo
- 手机窄宽下语音面板不会造成工具栏和提交按钮错位
- 有语音的 memo 卡片可正常显示播放器
- 普通 memo 的行为完全不变

### 后端

- 创建带 `voiceNote` 的 memo
- 查询 memo 时返回语音信息
- 创建语音记录时正确写入初始状态
- 异步转写任务能将状态从 `pending` 更新到 `done` 或 `failed`

### 集成

- 只有音频、无正文时的 memo 能正确展示
- 转写完成后正文能展示
- 转写失败时播放器仍可用

## 实施顺序建议

建议按以下顺序实现：

1. 数据结构和 migration
2. worker 路由扩展，支持创建和读取 `voiceNote`
3. 前端 `MemoComposer` 录音与保存流程
4. 时间线卡片和详情页播放器渲染
5. 异步转写状态结构与占位逻辑
6. 测试补齐

## 风险与取舍

主要风险：

- 浏览器录音能力在移动端兼容性不完全一致
- 浏览器原生识别不稳定，不能作为产品硬依赖
- 若 `content` 允许为空，需要确认现有查询和展示没有隐式依赖“内容非空”

主要取舍：

- 一期优先把语音保存链路做稳，而不是追求稳定转写
- 一期采用独立语音表，换取更清晰的数据边界和后续扩展能力
- 一期保持“正文仍是正文、音频是附件”的产品心智，不把音频序列化进 markdown

## 结论

一期实现采用以下方案：

- 编辑器新增语音按钮，位于图片按钮前
- 网页端支持录音、预览、重录、取消、保存
- 语音文件先上传，再创建 memo
- memo 通过独立 `voiceNote` 结构挂载音频元数据
- 时间线和详情页优先展示播放器
- 转写异步补齐，且必须可降级
- 没有转写能力时也必须能稳定保存语音 memo
